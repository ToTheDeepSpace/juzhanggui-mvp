> **版本**：BP v4.0  
> **日期**：2026-05-05  
> **状态**：两个产品均已上线，正在打通

---

## 一、两个产品，一套底层

### 剧掌柜 — 店家的操作系统

演绎剧场/密室/沉浸式戏剧的**排期 + 抢车 + 签到 + 管理**工具。

```
排期管理 → 抢车模式 → 玩家扫码 → 满员发车
卡司管理 → 房间管理 → 数据看板
```

**核心用户**：店家客服、店长  
**已上线**：`script-scheduler.vercel.app`

### 灵契 — 创作者的职业主页

卡司、Coser、摄影师、妆造师的**个人主页 + 档期展示 + 粉丝联系**平台。

```
个人主页 → 作品集 → 档期日历 → 可接服务 → 粉丝申请联系
```

**核心用户**：卡司/DM、Coser、摄影师、妆造师  
**已上线**：`lingqi.vercel.app`

### 底层已经打通

| | 剧掌柜 | 灵契 |
|--|--------|------|
| **数据库** | Supabase（`schedules`, `actors`, `checkins`） | **同一 Supabase**（`lc_profiles`, `lc_availability`）|
| **卡司数据** | 店家的卡司列表（`actors`）| 创作者的公开主页（`lc_profiles`）|
| **档期数据** | 排班表上的时段 | 创作者自己标记的可约日期 |
| **认证** | admin 密码登录 | 手机号 + 密码登录 |

## 二、打通方式（现在就可行）

### 场景一：卡司入驻灵契 → 自动关联剧掌柜

```
卡司在灵契注册 → 创建个人主页
              → 填写手机号
              → 剧掌柜自动识别该卡司
              → 排班时该卡司的数据直接同步
```

**现在的数据基础：** 剧掌柜的 `actors` 表和灵契的 `lc_profiles` 表可以通过手机号关联。

### 场景二：玩家通过灵契约卡司 → 直接对应剧掌柜的排期

```
粉丝在灵契看到卡司 → 查看档期
                    → 点击「申请联系」
                    → 客服收到申请
                    → 在剧掌柜创建排期
                    → 粉丝收到确认
```

**现在的数据基础：** 灵契的 `lc_contact_requests` 已经记录了粉丝的申请。

### 场景三：灵契的档期日历 ←→ 剧掌柜的排班表

```
剧掌柜创建排期 → 自动在对应卡司的灵契主页显示该时段不可约
             → 卡司不需要手动重复填档期
```

**现在的数据基础：** 剧掌柜的 `schedules` 表有 `actor_id`，灵契的 `lc_availability` 有 `creator_id`，可以通过 `actors.phone = lc_profiles.phone` 关联。

## 三、用户价值闭环

```
卡司（灵契）
   ↓ 有主页展示自己、接委托
   ↓ 在剧掌柜的店家演出
   ↓ 剧掌柜帮卡司管理排期、统计数据

店家（剧掌柜）
   ↓ 使用排期/抢车/签到系统
   ↓ 在灵契上发现好卡司
   ↓ 灵契帮店家吸引玩家

玩家（灵契 → 剧掌柜）
   ↓ 在灵契发现喜欢的卡司
   ↓ 查看卡司的演出档期
   ↓ 通过剧掌柜扫码上车、选角色
```

## 四、商业模式

### 收入来源

| 来源 | 说明 |
|------|------|
| **剧掌柜 SaaS 订阅** | 店家按月付费（¥299-999/月）|
| **灵契创作者增值** | 高级主页、数据分析（免费+增值模式）|
| **打通价值** | 卡司数据 + 档期数据 = 店家和创作者都离不开 |

### 竞争壁垒

1. **双边网络效应**：卡司越多 → 店家越多 → 玩家越多 → 卡司越多
2. **数据绑定**：排期数据 + 创作者数据 + 签到数据 = 竞争对手抄不走
3. **离线运营**：一家店一家店签，一场展一场展跑，互联网大厂不愿意干
